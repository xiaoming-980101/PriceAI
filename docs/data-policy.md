# PriceAI Data Policy Draft

本文档用于确认 PriceAI 下一阶段的数据策略。它覆盖两个当前最重要的问题：

1. 采集失败、重试、过期和前台有货/缺货展示逻辑。
2. 标准商品分类过细或不准的问题。

本文已作为当前数据逻辑的实现依据；阈值仍可继续按真实采集表现调整。

## 1. 目标

PriceAI 的核心不是维护一堆人工报价，而是尽量自动、稳定地从原站采集真实价格和库存，并把不同卡网的原始商品归到足够清晰、但不过细的标准商品里。

最终用户只需要快速知道：

- 哪个标准商品当前有货。
- 有货报价里的最低价是多少。
- 来自哪个渠道。
- 最近什么时候成功确认过。
- 点哪里去原站购买。

后台运营者需要快速知道：

- 哪些渠道采集成功。
- 哪些渠道采集失败。
- 为什么失败。
- 是否已经自动重试。
- 是否需要新增解析脚本或修复采集器。

## 2. 基本原则

### 2.1 前台状态只保留两个

前台面向用户只显示：

- 有货
- 缺货

不引入可信度、低可信、待确认、半确认、复合状态等复杂标签。

后台可以保留更多内部状态，例如采集失败、重试中、解析失败、待开发采集器、连续失败次数等，但这些是运营诊断信息，不应该直接变成前台复杂标签。

### 2.2 采集失败不等于立即缺货

一次采集失败只说明这次没有成功读取原站，不等于原站商品已经缺货。

因此不应该在单次采集失败后，立刻把该渠道所有旧报价都改成缺货。

正确方向是：

1. 先记录失败原因。
2. 自动重试。
3. 如果重试仍失败，进入后台采集健康问题。
4. 在超过约定时间或连续失败次数后，再把相关报价降级为前台缺货或不参与最低价。

### 2.3 采集成功但商品消失，可以认为该商品缺货或下架

如果某个来源本轮采集成功，并且采集结果是完整的，那么旧报价没有再出现时，可以将这条旧报价标为缺货或过期。

这里和采集失败不同：

- 采集失败：不知道原站状态，不能立刻判定缺货。
- 采集成功但商品没出现：可以认为原站已经不再提供该商品。

如果采集结果是部分成功，需要在后台标记 partial，不应直接批量改旧报价。

### 2.4 最低价只看有效有货报价

首页、列表页、标准商品卡片、标准商品表格里的最低价，只能来自有效有货报价。

缺货报价可以在详情页展示，但必须明显弱化，不能参与外层最低价。

## 3. 采集与状态模型

### 3.1 建议拆分两个时间

前台和后台应区分两个时间：

- 最近确认时间：该报价或来源最后一次成功采集并确认状态的时间。
- 最近检查时间：系统最后一次尝试采集该来源的时间，不管成功还是失败。

前台优先展示最近确认时间。后台需要同时展示最近确认时间和最近检查时间。

原因是：如果 7 分钟前检查失败，但 2 小时前成功确认有货，用户看到“7 分钟前有货”是不准确的；后台看到“7 分钟前检查失败”才是正确诊断信息。

### 3.2 有货判定草案

一条报价在前台显示为有货，需要同时满足：

- 原站最近一次成功采集时，该报价状态不是缺货。
- 这条报价仍在有效期内。
- 所属来源没有超过失败降级阈值。
- 报价有可用价格和购买链接。

如果不满足，则前台按缺货处理，或者不参与最低价。

### 3.3 采集失败后的处理流程

建议流程：

1. 对来源发起采集。
2. 如果成功，更新报价、库存、价格、最近确认时间，并清零连续失败次数。
3. 如果失败，记录失败原因和最近检查时间。
4. 自动重试，建议 2 到 3 次，间隔逐步拉长。
5. 如果重试成功，按成功流程处理。
6. 如果重试仍失败，后台显示采集异常，并增加连续失败次数。
7. 如果连续失败超过阈值，或距离最近成功确认超过阈值，则前台不再把旧有货报价当作有效有货。

### 3.4 失败阈值待确认

以下只是建议值，后续可以按实际情况调整：

- 短暂失败：1 到 2 次失败，只在后台提示，不改变前台旧状态。
- 需要关注：连续失败 3 次，后台高亮，支持一键重试。
- 前台降级：超过 12 到 24 小时没有成功确认，该来源旧报价不再参与最低价。
- 强制缺货：超过 24 到 48 小时没有成功确认，旧报价前台显示为缺货。

最终阈值需要再确认。

### 3.5 后台需要补充的能力

后台应围绕“保证采集成功”设计，而不是围绕人工改价设计。

建议增加：

- 单渠道重新采集按钮。
- 失败来源列表。
- 连续失败次数。
- 最近失败原因。
- 最近确认时间。
- 最近检查时间。
- 自动重试记录。
- 采集器待办：用于保存暂时不支持、但未来要写解析脚本的渠道。

人工补录只作为调试或临时兜底，不作为长期工作流。

## 4. 标准商品分类原则

当前问题不是分类不够细，而是分类层级混在一起：

- 平台混在标准商品里。
- 交付方式混在标准商品里。
- 邮箱交付导致 ChatGPT 普号落入邮箱。
- CDK/API 关键字过宽，导致 Gemini、Suno、Codex、API 中转混到同一类。
- `pro 20` 规则过宽，导致 Gemini Pro 2020-2024 被误判为 ChatGPT Pro 20x。

新的方向是：标准商品保持偏粗，先保证比价可用，再用标签表达细节。

## 5. 建议标准商品列表

### 5.1 ChatGPT

- ChatGPT Plus
- ChatGPT Plus 成品号
- ChatGPT Pro 5x
- ChatGPT Pro 20x
- ChatGPT Team / Business
- ChatGPT 普号 / Free 号

说明：

- ChatGPT Plus 官方直充、正规充值、自助开通、卡密、渠道开通，先合并到 ChatGPT Plus。
- Team 母号、邀请、自动拉、拼车位，先合并到 ChatGPT Team / Business。
- 普号、白号、Free 号应该归 ChatGPT，不应该因为使用微软邮箱交付而落入邮箱分类。

### 5.2 Claude

- Claude Pro
- Claude Max 5x
- Claude Max 20x
- Claude 普号 / 兑换号

说明：

- Claude Pro 直充、代充、激活码、卡密先合并。
- Claude Max 账号、礼品卡、代开先按 5x / 20x 分。

### 5.3 Gemini

- Gemini Pro
- Gemini Ultra / Google AI Ultra

说明：

- Gemini Pro 年卡、成品号、CDK、优惠链接、自助充值，先合并到 Gemini Pro。
- Gemini Ultra、Google AI Ultra、企业 Ultra、反重力、Flow 积分，先合并到 Gemini Ultra / Google AI Ultra。
- 教程、虚拟卡、接码这类不归 Gemini Pro，进入其他或辅助类。

### 5.4 Grok

- Super Grok
- Grok 普号 / 体验号

说明：

- Super Grok 成品号、卡密、直充、激活码、月卡、年卡先合并。
- 普号、体验号、短期直登号归 Grok 普号 / 体验号。

### 5.5 API / CDK / 额度

- API / CDK / 额度

说明：

- 这一阶段不细分 OpenAI API、Codex API、GPT API、中转余额、额度兑换等。
- 但是如果标题明确是 Gemini Pro CDK、Super Grok 激活码、ChatGPT Plus 卡密，应优先归对应平台标准商品，不能只因为出现 CDK 就归到 API。
- API 类更适合承接通用 API、中转、余额、额度、Codex API、OpenAI API 等商品。

### 5.6 邮箱

方案 A：先保留一个大类。

- 邮箱账号

方案 B：如果后续需要更清晰，可以拆成两个。

- Gmail / Google 邮箱
- Outlook / Hotmail / Microsoft 邮箱

当前建议先用方案 A，保持简单。

说明：

- 只有纯邮箱商品进入邮箱。
- ChatGPT 普号、OpenAI 普号、Claude 普号，即使通过微软邮箱或 Gmail 交付，也归到对应平台账号类。

### 5.7 其他

- 其他商品

说明：

- Cursor、Kiro、Suno、Windsurf、X Premium、Apple ID、Telegram、Facebook、接码、虚拟卡、代理服务、教程等，第一阶段都先进入其他商品。
- 后续如果某个其他品类数量变多、并且确实需要比价，再单独提升为标准商品。

## 6. 分类规则优先级

建议分类顺序如下：

1. 先识别明确平台：ChatGPT、Claude、Gemini、Grok。
2. 再识别明确套餐：Plus、Pro 5x、Pro 20x、Team、Claude Max、Gemini Ultra、Super Grok。
3. 再识别交付方式：成品号、普号、卡密、直充、兑换码、链接、账号。
4. 再识别 API / 中转 / 额度。
5. 再识别邮箱。
6. 最后进入其他。

关键修正：

- `CDK`、`卡密`、`兑换码` 只是交付方式，不应该天然归 API。
- `邮箱` 也只是交付方式或账号载体，不应该天然吞掉 ChatGPT 普号。
- `20x` 只有在标题同时明确出现 ChatGPT/GPT Pro 时，才可以归 ChatGPT Pro 20x。
- `母号` 只有在标题同时明确出现 ChatGPT/GPT Team/Business 时，才可以归 ChatGPT Team / Business。
- Gemini / Google AI 的规则应优先于 ChatGPT Pro 数字规则，避免 `Gemini Pro 2020-2024` 被误判。

## 7. 前台展示规则

### 7.1 首页和标准商品列表

- 显示标准商品。
- 最低价只取有效有货报价。
- 缺货报价数量可以显示，但不参与最低价。
- 其他商品可以显示，但不应该污染 ChatGPT、Gemini、API 等主类。

### 7.2 商品详情页

详情页展示所有报价，包括有货和缺货。

排序建议：

1. 有货报价在前。
2. 有货报价按价格从低到高。
3. 缺货报价在后，并弱化显示。
4. 采集过期或来源异常的旧报价，不参与可购买排序。

### 7.3 更新时间展示

前台建议文案：

- 有货报价：最近确认 7 分钟前。
- 缺货报价：最近确认缺货 7 分钟前。
- 来源采集异常时，不直接在前台扩展复杂状态，后台处理。

后台建议文案：

- 最近确认：2 小时前成功确认有货。
- 最近检查：7 分钟前检查失败。
- 连续失败：3 次。
- 失败原因：页面结构变化 / WAF / 请求超时 / 解析为空。

## 9. 下一步实施建议

确认本文档后，建议按以下顺序实现：

1. 调整分类目录，按本文标准商品列表重建 canonical catalog。
2. 重写分类规则，改成平台优先、交付方式其次、API 和邮箱靠后。
3. 对现有 352 条报价重新分类，并生成后台可查看的分类结果。
4. 增加采集失败重试和失败原因记录。
5. 增加来源健康后台视图。
6. 调整最低价逻辑，只使用有效有货报价。
7. 调整详情页排序，让缺货和过期报价明显靠后。
