# PriceAI 数据与分类重构方案

本文档是 [`docs/data-policy.md`](../data-policy.md) 的下一版细化草案，用于确认后续开发逻辑。它覆盖四件事：

1. 价格采集与解析准确性。
2. 标准商品分类规则。
3. 平台全部报价视图。
4. 用户提交渠道到采集器扩展的闭环。

本文档确认前不直接作为实现依据。确认后，建议把它拆成开发任务并逐步落地。

## 1. 背景与目标

当前 PriceAI 已经能自动采集多渠道报价，但出现了几个关键问题：

- 部分原站价格解析错误，例如 `¥1,280.00` 被解析成 `¥1`。
- 标准商品分类过细，Plus、Plus 成品号、卡密、直充、Team 等混在一起后容易误判。
- `其他商品` 中混入 ChatGPT、Grok、Gemini 等主平台商品。
- 用户提交渠道后，如果当前采集器不支持，还没有形成从待办到新增解析脚本再回到自动采集的完整闭环。

本轮重构目标：

- 价格必须先准，异常价格不能参与最低价。
- 分类要更粗、更稳定，优先服务比价，不追求过细。
- 前台仍只保留 `有货` 和 `缺货` 两个库存状态。
- 所有原始报价都能通过平台全部报价视图查到，避免分类错误时完全看不到。
- 渠道审核以自动采集为核心，人工补录不作为长期路径。

## 2. 核心术语

### 2.1 来源渠道

来源渠道指一个可采集的卡网或卡网店铺入口，例如：

- `https://aisou.pro/`
- `https://pay.ldxp.cn/shop/jinyao`
- `https://bei-bei.shop`
- `https://shop.auto-subscribe.com/`

来源渠道保存到 `sources`。只要来源启用，并且系统能识别采集器，定时任务就应该能自动跑到它。

### 2.2 原始报价

原始报价指从原站采集到的一条商品记录，保存到 `raw_offers`。它必须保留：

- 来源渠道。
- 原始商品名。
- 原始价格。
- 原始库存状态。
- 购买链接。
- 最近确认时间。
- 采集失败信息。
- 归类后的平台和标准商品。

原始报价是事实层，不能因为分类调整而丢失。

### 2.3 标准商品

标准商品是给用户比价用的聚合层。例如：

- ChatGPT Plus
- ChatGPT Pro 5倍
- ChatGPT Pro 20倍
- Gemini Pro
- Super Grok

标准商品应该偏粗。它不是完整商品目录，也不是所有交付方式的枚举。

### 2.4 平台

平台是比标准商品更上层的筛选维度，例如：

- ChatGPT
- Claude
- Gemini
- Grok
- API/CDK
- 邮箱
- 虚拟卡
- 其他

平台用于用户先缩小范围。标准商品用于进一步聚合比较。

### 2.5 未匹配标准商品

有些报价属于某个平台，但暂时无法确认具体标准商品。例如标题只写了 `ChatGPT 账号`，但没有明确普号、Plus、Pro 或 Team。它仍然应该出现在 `ChatGPT -> 全部报价` 中，但暂时不参与标准商品聚合。

建议内部使用 `未匹配标准商品` 或 `platform_unmapped` 这类概念，但前台不把它做成复杂标签。

## 3. 价格解析策略

### 3.1 当前问题

贝贝页面真实 HTML 中价格为：

```text
¥1,280.00
```

当前正则只支持：

```text
¥1280.00
```

所以只抓到逗号前的 `1`，最终显示成 `¥1`。

这类错误会直接污染最低价，必须优先修。

### 3.2 价格解析原则

价格解析应遵守以下原则：

1. 价格必须在商品卡片作用域内解析，不能跨商品卡片读取。
2. 优先读取价格专用节点，例如 `strong`、价格 class、API 字段。
3. 支持千分位，例如 `1,280.00`。
4. 支持人民币符号 `¥`、`￥`，以及纯数字 API 字段。
5. 不允许把销量、库存、规格编号、渠道编号当成价格。
6. 价格解析失败时，该报价不入库或不更新价格。
7. 价格解析失败时保留旧数据或记录失败，不用人为价格阈值判断。

### 3.3 解析格式

建议统一的价格解析函数支持：

```text
¥1,280.00 -> 1280
￥1,280 -> 1280
¥103.40 -> 103.4
103.40 -> 103.4
```

并明确拒绝：

```text
已售 100
库存 5
渠道1
规格2
20X
200刀款
```

### 3.4 不做价格阈值保护

第一版不做 `Pro 20倍低于多少就是异常` 这类保护。

原因：

- 卡网价格波动大，低价不一定是错误。
- 有些商品标题本身就可能混杂规格、地区、时长、质保规则，阈值容易误伤。
- 用户前台只需要知道这条报价当前是 `有货` 还是 `缺货`。

因此最低价只看两个条件：

1. 价格字段能被正确解析。
2. 报价状态是 `有货`。

如果价格解析失败，就不要用错误价格更新报价。后续排查重点应该是采集器解析是否准确，而不是给商品设置固定价格下限。

### 3.5 采集器测试要求

每个采集器都应有最小测试样例：

- 一条普通价格。
- 一条千分位价格。
- 一条缺货商品。
- 一条库存大于 0 的商品。
- 一条包含销量或渠道编号但不能误判为价格的商品。

贝贝采集器必须包含以下回归样例：

```text
ChatGPT Pro 20X 月卡｜官方卡充｜1个月｜支持续费｜正规充值
价格：¥1,280.00
期望解析：1280
```

## 4. 分类总体原则

### 4.1 分类不是越细越好

标准商品的目标是快速比价，不是完整商品知识库。

过细分类会带来三个问题：

- 标题表达不统一，容易误分。
- 同一个用户意图被拆散，无法比较最低价。
- 后台需要大量人工纠正。

本轮重构以粗分类为主。

### 4.2 标题优先，标签辅助

分类主判断只看原始商品标题和必要的商品描述。

来源标签、分类名、店铺栏目只能作为弱辅助，不能覆盖标题。例如：

```text
标题：GPT Plus 土区官方直充 -卡密自助
来源标签：team
```

最终应该归为 `ChatGPT Plus`，不能因为标签里有 `team` 就归到 Team / Business。

### 4.3 平台优先，但要识别“兼容提及”

很多辅助商品会在标题里提到平台，但核心商品不是这个平台。例如：

```text
VISA 0刀虚拟卡 485954（gemini，gpt需要自测）
```

这里核心商品是虚拟卡，不是 Gemini 或 ChatGPT。

因此规则要区分：

- 核心商品信号：商品本体是什么。
- 兼容提及：这个商品可用于哪个平台。

核心商品信号优先。

### 4.4 交付方式不是标准商品

以下词通常是交付方式，不应该单独变成标准商品：

- 成品号
- 直充
- 代充
- 卡密
- CDK
- 激活码
- 自助
- 链接
- 邀请
- 账号密码
- JSON
- 首登

这些词可以保存为标签，但不应该把 `ChatGPT Plus` 拆成多个标准商品。

### 4.5 API、邮箱、虚拟卡靠后但有核心词例外

`CDK` 不等于 API。`邮箱` 不等于邮箱商品。`虚拟卡` 有时只是使用说明。

规则：

- 标题明确是 `ChatGPT Plus 卡密`，归 ChatGPT Plus。
- 标题明确是 `Gemini Pro CDK`，归 Gemini Pro。
- 标题明确是 `Super Grok 激活码`，归 Super Grok。
- 标题核心是 `OpenAI API`、`中转余额`、`Codex API`，归 API/CDK。
- 标题核心是 `Gmail`、`Outlook`、`微软邮箱`，且没有主平台套餐，归邮箱。
- 标题核心是 `VISA`、`0刀卡`、`虚拟信用卡`，归虚拟卡。

## 6. 分类决策流程

### 6.1 标准化输入

分类前先做标准化：

- 全部转小写。
- 中文全角符号转空格。
- `20×` 规范成 `20x`。
- `5×` 规范成 `5x`。
- `美金`、`美元`、`刀` 保留。
- 常见错拼归一：
  - `puls` -> `plus`
  - `pulus` -> `plus`
  - `chat gpt` -> `chatgpt`
  - `supergrok` -> `super grok`

### 6.2 核心商品识别

第一步识别核心商品，不直接看弱标签。

优先级建议：

1. 虚拟卡核心词：`visa`、`mastercard`、`虚拟卡`、`0刀卡`、`1刀卡`、`485954`。
2. API 核心词：`api 中转`、`中转站`、`余额兑换`、`codex api`、`openai api`。
3. 明确平台套餐：ChatGPT、Claude、Gemini、Grok。
4. 纯邮箱：Gmail、Outlook、Hotmail、微软邮箱。
5. 其他工具：Kiro、Cursor、Suno、Windsurf、X Premium。
6. 兜底其他。

注意：如果标题核心是 `ChatGPT Plus`，即使包含 `卡密`、`CDK`、`邮箱`，也应该先归 ChatGPT。

### 6.3 ChatGPT 细分流程

当标题被识别为 ChatGPT 后，按以下顺序细分：

1. 如果命中 `pro 20x`、`20倍`、`x20`、`200刀`、`200美元`、`200美金`，归 `ChatGPT Pro 20倍`。
2. 如果命中 `pro 5x`、`5倍`、`x5`、`100刀`、`100美元`、`100美金`，归 `ChatGPT Pro 5倍`。
3. 如果命中 `plus`、`plus 月卡`、`plus 成品号`、`plus 直充`、`plus 卡密`、`土区 plus`、`ios plus`，归 `ChatGPT Plus`。
4. 如果命中 `team`、`business`、`t5`、`t5倍`、`母号`、`邀请`、`自动拉`，归 `ChatGPT Team / Business`。
5. 如果命中 `free`、`普号`、`白号`、`普通账号`、`openai 普号`，归 `ChatGPT 普号`。
6. 其余 ChatGPT 相关报价进入 `ChatGPT 全部报价`，但不参与标准商品聚合，等待人工确认或后续规则补充。

注意：`Plus` 优先级高于弱来源标签。只有标题本身明确表达 Team / Business 时，才归 `ChatGPT Team / Business`。

### 6.4 置信度只用于后台

分类可以在后台生成内部置信度：

- `high`：标题明确，自动应用。
- `medium`：标题有平台和套餐，但存在噪声，自动应用并可在后台筛查。
- `low`：只知道平台，不确定标准商品，进入平台全部报价，不进入标准商品。

前台不显示置信度。

### 6.5 分类覆盖规则

后续后台应支持人工覆盖分类，但覆盖应该保存为规则或明确记录，不应只是改一次数据库。

建议覆盖类型：

- 单条报价覆盖：只影响当前报价。
- 来源规则覆盖：该来源某个分类名或标签固定映射。
- 标题关键词规则：命中固定关键词后映射到某标准商品。

## 7. 平台全部报价视图

### 7.1 为什么需要

标准商品分类永远不可能 100% 准。平台全部报价视图的价值是：

- 用户可以绕过标准商品分组，直接看某个平台所有原始报价。
- 后台也可以用它发现误分类。
- 对暂时无法进入标准商品的报价，仍然有展示位置。
- 用户想粗看 ChatGPT 市场时，不必先判断自己要点 Plus、Pro、Team 还是普号，可以直接扫一整个平台的报价。

### 7.2 入口位置

建议在平台选择后出现视图切换：

```text
ChatGPT

[标准商品] [全部报价]
```

默认：

- 首页默认 `标准商品`。
- 选择具体平台后仍默认 `标准商品`。
- 用户切到 `全部报价` 后，在同一会话保持选择。

### 7.3 标准商品视图

继续展示当前聚合列表，重点是：

- 标准商品名。
- 平台。
- 最低有货价。
- 有货数。
- 缺货数。
- 渠道数。
- 最低价来源。
- 最近确认时间。

最低价只来自有效有货报价。

### 7.4 全部报价视图

桌面端建议表格：

| 状态 | 渠道 | 原始商品名 | 价格 | 最近确认 | 操作 |
| --- | --- | --- | --- | --- | --- |
| 有货 | beibei | ChatGPT Pro 20X 月卡... | ¥1280 | 6 分钟前 | 前往购买 |
| 有货 | GPT专卖-cw | GPT Plus 土区官方直充 -卡密自助 | ¥128 | 9 分钟前 | 前往购买 |
| 缺货 | Auto Subscribe | Business Team 母号 / 规格2 | ¥1.5 | 8 分钟前 | 查看 |

这个视图的含义是：选中 `ChatGPT` 后，把所有属于 ChatGPT 平台的原始报价全部展示出来，包括：

- ChatGPT 普号。
- ChatGPT Plus。
- ChatGPT Team / Business。
- ChatGPT Pro 5倍。
- ChatGPT Pro 20倍。
- 暂时只能确认是 ChatGPT、但还没匹配到具体标准商品的报价。

它更像“平台级详情页”，不是后台分类表。因此前台不需要展示 `归类` 这一列。

排序默认：

1. 有货在前。
2. 有货报价按价格从低到高。
3. 缺货报价排在后面。
4. 同状态同价格时，最近确认时间新的在前。

筛选建议：

- 搜索原始商品名。
- 有货 / 缺货。
- 来源渠道。
- 价格区间。

### 7.5 移动端

移动端不使用宽表格，改成紧凑列表：

```text
有货  ¥1280
beibei
ChatGPT Pro 20X 月卡｜官方卡充｜1个月...
最近确认 6 分钟前
[前往购买]
```

缺货报价使用浅红底或浅红状态 chip，但仍显示原始价格，避免丢失信息。

### 7.6 与详情页的关系

标准商品详情页仍保留，但全部报价视图是平台级详情。

- 标准商品详情页：只看某个标准商品的所有渠道。
- 平台全部报价视图：看某个平台下所有原始报价，效果类似把该平台下多个标准商品详情页合并到一张总表。

这两个视图解决不同问题，建议都保留。

## 8. 大模型辅助分类方案

### 8.1 是否立刻引入

建议不作为第一阶段核心依赖。先把规则和数据结构调稳。

大模型适合做：

- 低置信度分类建议。
- 其他商品中的疑似主平台商品识别。
- 新渠道采集后的初次分类审查。
- 规则回归样本生成。

不适合做：

- 每次前台请求实时分类。
- 直接覆盖人工确认规则。
- 无记录地修改历史数据。

### 8.2 输入

大模型分类输入建议包括：

```json
{
  "sourceName": "beibei",
  "sourceTitle": "ChatGPT Pro 20X 月卡｜官方卡充｜1个月｜支持续费｜正规充值",
  "price": 1280,
  "tags": ["ChatGPT", "推理强"],
  "currentPlatform": "ChatGPT",
  "currentCanonicalProduct": "chatgpt-pro-20x",
  "allowedProducts": ["ChatGPT 普号", "ChatGPT Plus", "ChatGPT Team / Business", "ChatGPT Pro 5倍", "ChatGPT Pro 20倍"]
}
```

### 8.3 输出

只允许结构化 JSON：

```json
{
  "platform": "ChatGPT",
  "canonicalProduct": "ChatGPT Pro 20倍",
  "confidence": "high",
  "reason": "标题明确包含 ChatGPT Pro 20X",
  "needsHumanReview": false
}
```

### 8.4 应用方式

建议第一版只在后台使用：

1. 规则分类先跑。
2. 找出低置信度或可疑报价。
3. 批量调用大模型生成建议。
4. 后台显示建议和原因。
5. 管理员确认后写入分类覆盖或规则。

确认前不影响前台。

## 9. 渠道提交与采集器闭环

### 9.1 当前理想流程

用户提交渠道后，系统应自动进入以下流程：

```text
用户提交 URL
  -> 初步解析
  -> 后台待审核
  -> 试采集
  -> 成功：审核通过并加入 sources
  -> 失败但真实：加入采集器待办
  -> 不相关或无效：拒绝
```

### 9.2 初步解析

提交时立即解析：

- URL 是否合法。
- 域名是否公开可访问。
- 是否疑似重复来源。
- 来源名称建议。
- 来源 ID 建议。
- 采集器类型建议，例如 `kami`、`dujiao`、`shopApi`、`beibeiHtml`、`unsupported`。
- 是否命中已支持 host。

初步解析不等于通过，只是减少审核负担。

### 9.3 试采集

审核页必须有 `试采集` 按钮。

试采集成功的判断：

- 能识别采集器。
- 能拿到至少一条有效报价。
- 报价有标题、价格、状态、购买链接。

试采集结果应展示：

- 采集器类型。
- 报价数量。
- 样例报价 5 到 10 条。
- 失败原因。
- 建议下一步。

### 9.4 审核通过

只有以下情况允许审核通过：

1. 匹配到已有来源，并且试采集成功。
2. 新来源命中已有采集器，并且试采集成功。

通过后执行：

- 写入或合并 `sources`。
- 写入试采集报价。
- 设置 `enabled = true`。
- 保存 `collection_method` 和 `collector_kind`。
- 下一次 GitHub Actions 定时采集时自动包含该来源。

也就是说，只要通过审核，不需要手动补价格。

### 9.5 加入采集器待办

如果渠道是真实的，但当前无法解析，不应该长期人工补录。

应进入 `采集器待办`，待办信息包括：

- URL。
- 域名。
- 提交人备注。
- 初步解析结果。
- 试采集失败原因。
- 需要的采集器类型建议。
- 是否可能属于已有采集器家族。
- 最近一次试采集时间。
- 失败日志摘要。

待办状态建议：

| 状态 | 含义 |
| --- | --- |
| 待分析 | 已确认真实渠道，但还没判断采集器类型 |
| 待开发采集器 | 需要新增解析脚本 |
| 已开发待验证 | 代码已支持，等待重新试采集 |
| 可通过 | 重新试采集成功，可审核通过 |
| 放弃 | 渠道长期无法公开采集或质量不适合 |

### 9.6 采集器扩展闭环

理想闭环如下：

```text
采集器待办
  -> 管理员确认真实渠道
  -> 生成开发任务上下文
  -> 开发新增解析脚本
  -> 部署
  -> 回到后台重新试采集
  -> 成功后审核通过
  -> 下次定时采集自动运行
```

后台第一版可以先提供 `复制开发上下文` 按钮，复制内容包括：

```text
请为 PriceAI 新增采集器支持：
- 来源 URL：
- 失败原因：
- 当前识别类型：
- 页面样例：
- 期望输出字段：sourceTitle, price, status, url, stockCount
- 验证方式：npm run collect:prices -- --source <source-id>
```

这样你不需要在对话里重新描述上下文，只要把待办复制给开发者或 Codex 即可。

后续增强可以考虑：

- 一键创建 GitHub Issue。
- 保存 HTML 快照或截图。
- 采集器模板生成。
- 对同域名多个提交自动归并。

### 9.7 拒绝

以下情况应拒绝：

- URL 无效或无法公开访问。
- 明显不是卡网或 AI 订阅相关渠道。
- 重复垃圾提交。
- 需要登录、验证码或绕过限制才可访问，并且不适合浏览器半自动采集。
- 渠道质量明显不适合纳入。

## 10. 后台页面调整建议

### 10.1 审核页

审核卡片建议展示：

- URL 和域名。
- 推荐渠道名。
- 是否匹配已有来源。
- 推荐采集器。
- 试采集状态。
- 样例报价。
- 下一步按钮。

按钮：

- `试采集`
- `通过并启用`
- `合并到已有来源`
- `加入采集器待办`
- `拒绝`

其中 `通过并启用` 在试采集成功前禁用。

### 10.2 采集器待办页

待办页按状态分组：

- 待分析。
- 待开发采集器。
- 已开发待验证。
- 可通过。
- 放弃。

每条待办提供：

- 重新试采集。
- 修改待办状态。
- 复制开发上下文。
- 标记已开发待验证。
- 通过并启用。
- 放弃。

### 10.3 来源渠道页

来源渠道页补充：

- 采集器类型。
- 最近确认时间。
- 最近检查时间。
- 连续失败次数。
- 最近失败原因。
- 单渠道重新采集。
- 查看该来源全部报价。

## 11. 数据结构建议

### 11.1 短期不强制大迁移

第一阶段可以继续使用现有表：

- `sources`
- `raw_offers`
- `canonical_products`
- `channel_submissions`
- `crawl_runs`

但需要补充几个内部字段或约定。

### 11.2 raw_offers 建议字段

建议新增或明确使用：

- `platform_slug`：平台，例如 `chatgpt`、`grok`、`virtual-card`。
- `canonical_product_id`：标准商品，可为空或为内部未匹配 ID。
- `classification_confidence`：后台内部字段。
- `classification_reason`：后台内部字段。
- `price_parse_status`：`ok`、`failed`。
- `price_parse_reason`：解析失败原因。
- `last_confirmed_at`：该报价最近一次被成功采集确认的时间。

如果短期不想加字段，可以先在 `details` 或后台逻辑中实现，但长期建议结构化。

### 11.3 canonical_products 调整

需要新增或调整：

- 合并 `chatgpt-plus-month` 和 `chatgpt-plus-account` 为 `chatgpt-plus`。
- 保留 `chatgpt-pro-5x`。
- 保留 `chatgpt-pro-20x`。
- 保留 `chatgpt-team-business`，名称统一为 `ChatGPT Team / Business`。
- 保留或重命名 `chatgpt-free-account` 为 `chatgpt-free`。
- 新增 `virtual-card`。

迁移时需要保留旧 ID 到新 ID 的映射：

| 旧 ID | 新 ID |
| --- | --- |
| chatgpt-plus-month | chatgpt-plus |
| chatgpt-plus-account | chatgpt-plus |
| chatgpt-team-business | chatgpt-team-business |
| other-product 中的 Super Grok | super-grok |
| other-product 中的虚拟卡 | virtual-card |

## 12. 开发顺序建议

### 阶段 1：价格准确性

目标：先修最伤信任的价格错误。

任务：

- 修正通用价格解析函数，支持千分位。
- 修正贝贝采集器。
- 修正浏览器采集价格正则。
- 对贝贝重新采集并覆盖错误报价。

验收：

- `ChatGPT Pro 20X` 贝贝报价显示 `¥1280`。
- 不再把 `¥1,280.00` 误解析为 `¥1`。

### 阶段 2：分类规则

任务：

- 重建 ChatGPT 标准商品目录。
- 合并 Plus 和 Plus 成品号。
- 调整标题优先、标签辅助逻辑。
- 增加虚拟卡平台。
- 修复 Grok、Gemini、ChatGPT 落入其他的问题。
- 对现有报价全量重新分类。

验收：

- `GPT Plus 土区官方直充 -卡密自助` 归 ChatGPT Plus。
- `GPT Plus 成品号` 归 ChatGPT Plus。
- `Super Grok` 不进入其他。
- `VISA 0刀虚拟卡` 进入虚拟卡。
- `OpenAI 普号【微软邮箱】` 归 ChatGPT 普号。

### 阶段 3：平台全部报价视图

任务：

- 在平台页增加 `标准商品 / 全部报价` 切换。
- 新增平台全部报价表。
- 支持来源、状态、价格区间筛选。
- 移动端改成紧凑列表。

验收：

- 进入 ChatGPT 后可直接查看所有 ChatGPT 原始报价。
- ChatGPT 普号、Plus、Team / Business、Pro 5倍、Pro 20倍都能在 ChatGPT 全部报价中看到。
- 缺货报价明显弱化。

### 阶段 4：渠道审核闭环

任务：

- 强化提交后的初步解析展示。
- 试采集成功前禁用通过。
- 待办页增加状态流转。
- 增加复制开发上下文。
- 重新试采集成功后可直接通过并启用。

验收：

- 新渠道从提交到通过能进入下次定时采集。
- 不支持的真实渠道能进入采集器待办。
- 采集器开发完成后，后台能重新试采集并通过。

### 阶段 5：大模型辅助分类

本阶段暂不进入第一期开发，只作为后续方向记录。

任务：

- 低置信度报价批量生成分类建议。
- 后台展示建议、原因、置信度。
- 人工确认后写入覆盖规则。

验收：

- 大模型结果不直接影响前台。
- 所有自动建议都可追溯。
- 能显著减少其他分类中的误入商品。
