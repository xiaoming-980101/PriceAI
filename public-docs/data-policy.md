# 数据策略

PriceAI 的核心不是维护一堆人工报价，而是尽量稳定地从公开或已授权来源采集真实价格、库存和来源信息，并把不同渠道的原始商品整理到足够清晰、不过度细碎的标准对象里。

## 面向用户的问题

用户进入 PriceAI 时，主要想快速回答：

- 这个商品、模型或站点现在是否可用？
- 当前可用报价里的最低价是多少？
- 价格来自哪个来源？
- 最近什么时候确认过？
- 点哪里回到原站核验？

## 前台状态

前台面向普通用户只保留清晰状态：

- 有货
- 缺货

后台可以保留采集失败、重试中、解析失败、待开发采集器、连续失败次数等诊断信息，但这些不应该直接变成复杂的前台标签。

## 采集失败不等于缺货

一次采集失败只说明这次没有成功读取原站，不等于原站商品已经缺货。

正确处理顺序：

1. 记录失败原因。
2. 按策略重试。
3. 如果持续失败，进入后台采集健康问题。
4. 超过约定时间或连续失败阈值后，再让旧报价退出外层最低价或降级展示。

## 最低价规则

首页、列表页、标准商品卡片和标准商品表格里的最低价，只能来自有效有货报价。

缺货、隐藏、未审核、过期或异常报价可以在合适的详情视图中保留，但不能作为外层最低价。

## 来源透明

公开展示应尽量保留：

- 原始来源名称。
- 原始商品标题。
- 原始价格和币种。
- 库存或可用状态。
- 最近确认时间。
- 原站链接。

PriceAI 不替渠道做担保，用户仍需要回到原平台确认最终价格、交付方式、售后规则和风险。

## 分类原则

分类目标是让用户能比较，而不是把每一种文案都拆成新商品。

例如，`直充`、`代充`、`成品号`、`卡密`、`CDK`、`Team 邀请` 通常是交付方式或风险信息，不应轻易变成完全独立的标准商品。标准商品应优先围绕用户真正要买的对象，例如 `ChatGPT Plus`、`Claude Pro`、`Gemini Pro`、具体官方 API 模型或具体中转站模型分组。
